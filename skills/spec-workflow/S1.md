# S1 产品语义事实源维护规则

## 1. 定位

S1 是当前领域产品语义的唯一权威事实源。

S1 是文档地位，不是固定文档格式。

任何能够完整、清晰、无冲突地描述产品目标、核心对象、业务规则、关键流程和边界的设计文档，都可以直接作为 S1。

## 2. 文件

每个领域只维护一个核心 S1 文件：

`00_product/domains/<domain_id>/product-spec.md`

已有设计文档可以直接迁移或重命名为该文件，不要求重新生成。

## 3. 保真原则

修订已有设计文档时，必须优先保留：

- 原章节结构；
- 原叙事顺序；
- 产品设计原因；
- 核心对象定义；
- 配置示例；
- 算法步骤；
- 端到端流程；
- Mermaid 图；
- 异常和恢复说明。

不得为了套用统一模板、编号体系、实体表格、用户故事或功能矩阵而压缩、拆散或删除已有产品事实。

## 4. 修改范围

默认只进行以下修改：

- 统一冲突术语；
- 删除语义重复；
- 修正章节和编号；
- 消除前后矛盾；
- 标记已废弃方案；
- 补充缺失的关键对象关系；
- 补充缺失的端到端链路；
- 补充关键校验与失败结果；
- 将具体代码实现改写为产品职责和语义契约。

除非原文已经无法顺序理解，或用户明确要求，不得全文重构。

## 5. S1 与 S2 边界

S1 保留：

- 产品能力清单；
- 种子数据；
- 逻辑字段；
- 配置契约；
- 对象关系；
- 适配器和执行器职责；
- 运行算法；
- 外部平台交互语义；
- 完整配置与运行示例；
- 关键异常与恢复策略。

S2 负责：

- 内部 HTTP 路径和方法；
- OpenAPI DTO；
- SQL 表、列、索引和外键；
- 错误码数字；
- 权限码数字；
- 具体编程语言接口；
- ORM 和代码结构。

## 6. 补充原则

发现原文缺失产品事实时，应将补充内容插入最相关的原章节。

不得仅因为信息缺失就全部转换成待确认问题。

只有存在真实产品决策冲突、多个方案均合理且无法从上下文判断时，才记录为待确认问题。

## 7. 可选结构

以下形式均可按需使用，但不强制：

- BR 业务规则编号；
- PF 流程编号；
- 用户故事；
- 验收标准；
- 功能矩阵；
- 对象分类表；
- 追溯矩阵；
- Mermaid 图。

只有当它们提高可理解性、可引用性或可验收性时才应加入。

## 8. 质量标准

一份 S1 合格的标准不是是否符合固定模板，而是：

- 人类可以顺序阅读并理解产品如何工作；
- 已确认的产品细节没有在整理过程中丢失；
- 同一概念在全文中含义一致；
- 主要业务链路完整闭环；
- 关键失败结果明确；
- S2 不需要自行发明核心产品语义。

## 9. 禁止行为

禁止：

- 将已有设计全文抽取后按统一模板重新生成；
- 将连续流程拆散成实体表、规则表和追溯矩阵；
- 将具体有效的配置示例压缩为抽象字段列表；
- 将原文已有事实重新标记为待确认；
- 为满足格式要求生成大量无价值编号；
- 因为内容接近技术而删除其产品语义；
- 生成比原文更短但更难理解的 S1；
- 让形式一致性优先于语义完整性。
