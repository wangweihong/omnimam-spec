# S2 实现契约事实源规则

S2 是实现契约事实源，是 S1 的结构化表达。

S2 用于定义可实现、可校验、可生成的接口、数据、错误码、权限码、事件和模块边界。

---

## 1. S2 定位

S2 必须从 S1 推导，不能凭空创造 S1 中未定义的核心业务能力。

S2 负责定义：

```text
API
请求响应结构
设计态 SQL schema
错误码
权限码
事件契约
模块契约
```

S2 不负责：

```text
正式实现代码
实际数据库 migration
前端组件实现
后端 service 函数实现
运行时配置
```

---

## 2. 文件位置

```text
01_contracts/
├── error-code-index.md
└── domains/
    └── <domain_id>/
        ├── openapi.yaml
        ├── schema.sql
        ├── errors.yaml
        ├── permissions.yaml
        ├── events.yaml
        └── module-contract.md
```

其中：

```text
error-code-index.md
  全局错误码索引与区间登记文件。

openapi.yaml
  API 契约。

schema.sql
  设计态 SQL schema。

errors.yaml
  当前 domain 的错误码定义。

permissions.yaml
  当前 domain 的权限码定义。

events.yaml
  当前 domain 的事件契约。

module-contract.md
  当前 domain 的模块边界与协作契约。
```

---

## 3. S1 追溯性要求

所有 S2 契约必须能追溯到 S1 元素。

必须显式引用相关 US/BR 的内容：

```text
OpenAPI operation
错误码
权限码
事件
模块契约
```

`schema.sql` 至少在表级标注来源，关键字段可补充字段级注释。

推荐引用字段：

```yaml
x-s1-refs:
  user_stories:
    - US-AICHAT-001
  business_rules:
    - BR-AICHAT-002
```

或使用等效字段：

```yaml
related_user_stories:
  - US-AICHAT-001
related_rules:
  - BR-AICHAT-002
```

要求：

```text
引用编号必须真实存在。
引用编号必须属于当前 domain 或明确依赖的 domain。
不得引用不存在、废弃或语义无关的编号。
```

## 4. API 契约规则

S2 中所有后端 API endpoint 必须遵循统一路径、方法、参数和响应规范。

---

### 4.1 API 路径前缀

所有后端 API endpoint 必须使用：

```text
/api/v1
```

示例：

```text
GET    /api/v1/providers
POST   /api/v1/providers
PATCH  /api/v1/providers/{provider_id}
DELETE /api/v1/providers/{provider_id}
POST   /api/v1/providers/{provider_id}/test
```

如确实存在不能使用 `/api/v1` 前缀的特殊场景，必须在 S2 中说明原因，并提示用户确认是否允许。

禁止未经确认直接定义非 `/api/v1` 路径。

---

### 4.2 HTTP Method 语义

API method 必须遵守以下语义：

```text
GET:
  查询单个资源、资源列表、只读聚合结果。

POST:
  创建资源，或触发动作类操作。

PATCH:
  局部更新资源。

PUT:
  整体保存或整体替换配置类资源。

DELETE:
  删除资源。
```

动作类接口必须使用：

```text
POST /api/v1/<resources>/{id}/<action>
```

示例：

```text
POST /api/v1/providers/{provider_id}/test
POST /api/v1/tasks/{task_id}/cancel
POST /api/v1/workflows/{workflow_id}/run
```

禁止使用 `GET` 触发有副作用的动作。

## 5. 请求参数规则

---

### 5.1 字段命名

请求参数字段必须使用 `lower_snake_case`。

示例：

```text
id
name
enabled
description
created_at
updated_at
provider_id
topic_id
```

禁止在 API 契约中混用：

```text
camelCase
PascalCase
kebab-case
```

除非外部第三方协议强制要求，否则 S2 中不得新增非 `lower_snake_case` 字段。

---

### 5.2 列表查询参数

列表查询接口应统一支持分页参数。

分页从 `0` 开始，`page_num = 0` 表示第一页。

通用列表查询参数如下：

| 字段             | 类型                | 默认值   | 说明                    |
| -------------- | ----------------- | ----- | --------------------- |
| page_num       | integer           | 0     | 分页页码，0 表示第一页          |
| page_size      | integer           | 由接口定义 | 每页数量                  |
| keyword        | string            | ""    | 模糊搜索关键词               |
| search_fields   | string            | "name,description"    | 模糊搜索字段,逗号分隔多个字段               |
| sort_field     | string            | ""    | 排序字段；为空时使用接口默认排序。多字段      |
| sort_order     | enum: asc, desc   | asc   | 排序方向                  |
| created_after  | string(date-time) | 无     | 筛选创建时间晚于该时间的资源        |
| created_before | string(date-time) | 无     | 筛选创建时间早于该时间的资源        |

业务特有筛选字段可以按需补充，但不得替代上述通用字段的语义。
如果新增字段需要传递数组，优先使用字符串类型定义，并通过","来分割，由后端实现切割成数组
---

### 5.3 批量请求参数

批量请求不得直接传递字符串数组或 ID 数组。

禁止：

```json
["provider_001", "provider_002"]
```

批量请求必须使用对象数组，并建议使用 `items` 包装，便于后续扩展字段。

推荐：

```json
{
  "items": [
    {
      "id": "provider_001"
    },
    {
      "id": "provider_002"
    }
  ]
}
```

当业务需要扩展时，可在单项对象中增加字段：

```json
{
  "items": [
    {
      "id": "provider_001",
      "reason": "manual cleanup"
    }
  ]
}
```

## 6. 响应参数规则

---

### 6.1 字段命名

响应参数字段必须使用 `lower_snake_case`。

示例：

```text
id
name
enabled
description
created_at
updated_at
total
items
```

---

### 6.2 普通成功响应

普通 JSON DTO 成功响应统一使用 HTTP `200`。

成功时直接返回业务对象，不强制包装为：

```json
{
  "code": 0,
  "data": {}
}
```

单对象响应示例：

```json
{
  "id": "provider_001",
  "name": "OpenAI Compatible",
  "enabled": true,
  "created_at": "2026-07-05T12:00:00Z",
  "updated_at": "2026-07-05T12:10:00Z"
}
```

---

### 6.3 列表响应

列表响应必须包含：

```text
total
items
```

示例：

```json
{
  "total": 2,
  "items": [
    {
      "id": "topic_001",
      "title": "Demo topic",
      "created_at": "2026-07-05T12:00:00Z",
      "updated_at": "2026-07-05T12:10:00Z"
    }
  ]
}
```

---

### 6.4 批量响应

批量接口 HTTP `200` 仅表示请求已被服务端处理，不代表每一项都成功。

批量响应必须提供逐项结果判断能力。

推荐字段：

```text
total
success
fail
results
```

其中 `results` 中每一项必须能表达该项成功或失败。

推荐结构：

```json
{
  "total": 2,
  "success": 1,
  "fail": 1,
  "results": [
    {
      "id": "topic_001",
      "success": true,
      "data": {
        "id": "topic_001",
        "title": "Demo topic",
        "created_at": "2026-07-05T12:00:00Z",
        "updated_at": "2026-07-05T12:10:00Z"
      }
    },
    {
      "id": "topic_002",
      "success": false,
      "error": {
        "code": 110201,
        "message": "Topic does not exist or does not belong to the current user.",
        "messages": {
          "zh-CN": "话题不存在或不属于当前用户。",
          "en-US": "Topic does not exist or does not belong to the current user."
        },
        "detail": "topic_id=topic_002 is not visible to current user",
        "causes": [],
        "data": {
          "id": "topic_002"
        }
      }
    }
  ]
}
```

批量接口不得只返回整体成功或失败而丢失单项状态。

## 7. OpenAPI 规则

`openapi.yaml` 覆盖所有需要服务支持的 S1 用户故事。

文件位置：

```text
01_contracts/domains/<domain_id>/openapi.yaml
```

必须定义：

```text
/api/v1 路径
HTTP 方法
请求参数
请求体
响应结构
错误响应
状态码
鉴权要求
分页规则
排序规则
过滤规则
x-s1-refs
```

OpenAPI 必须遵守本文件中的 API 路径、HTTP method、请求参数和响应参数规则。

禁止：

```text
先实现后反推 OpenAPI。
OpenAPI 与 S1 用户故事长期不一致。
OpenAPI 中出现 S1 未定义的核心业务能力。
不经用户确认使用非 /api/v1 路径。
```

## 8. SQL Schema 通用资源规则

---

### 8.1 schema.sql 定位

`schema.sql` 是设计态表结构，不是实际 migration。

文件位置：

```text
01_contracts/domains/<domain_id>/schema.sql
```

必须定义：

```text
表
字段
字段类型
主键
唯一约束
外键关系
索引建议
默认值
软删除策略
时间字段
JSON 字段约束
状态字段取值
S1 来源注释
通用资源元数据字段
```

建议用 SQL 注释标注来源：

```sql
-- s1_refs: US-AICHAT-001, BR-AICHAT-002
CREATE TABLE ai_assistants (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL,
  description TEXT DEFAULT '',
  extend_shadow TEXT DEFAULT ''
);
```

---

### 8.2 通用资源元数据字段

所有资源表默认必须包含通用资源元数据字段。

```sql
id TEXT PRIMARY KEY,
name TEXT NOT NULL,
created_at TIMESTAMPTZ NOT NULL,
updated_at TIMESTAMPTZ NOT NULL,
description TEXT DEFAULT '',
extend_shadow TEXT DEFAULT ''
```

字段语义：

| 字段            | 说明                      |
| ------------- | ----------------------- |
| id            | 资源唯一标识                  |
| name          | 资源名称                    |
| created_at    | 创建时间                    |
| updated_at    | 更新时间                    |
| description   | 资源描述                    |
| extend_shadow | 扩展影子字段，用于存储不适合单独建列的扩展数据 |

如果某类表不适合使用通用资源字段，必须在 S2 中说明原因，并提示用户确认是否允许。

---

### 8.3 extend_shadow 规则

`extend_shadow` 的数据库类型固定为：

```sql
TEXT
```

`extend_shadow` 存储 JSON 编码后的字符串，但不要求固定 JSON schema。

示例：

```json
{
  "1": {
    "inputs": {
      "text": "low quality",
      "clip": ["4", 1]
    },
    "class_type": "CLIPTextEncode",
    "_meta": {
      "title": "negative prompt"
    }
  },
  "2": "what enve"
}
```

适合放入 `extend_shadow`：

```text
低频访问的扩展字段
实验性字段
第三方 provider 原始配置
工作流节点原始片段
不稳定结构的附加信息
```

不适合放入 `extend_shadow`：

```text
高频查询字段
排序字段
权限判断字段
状态流转字段
需要索引的字段
强一致校验字段
```

资源表可以在通用字段基础上增加业务字段。

当字段需要查询、排序、过滤、校验或参与核心业务规则时，应独立建列，不应放入 `extend_shadow`。

---

### 8.4 schema.sql 禁止事项

禁止：

```text
将实际 migration 当作 S2 schema 事实源。
资源表缺少通用资源元数据字段且未说明原因。
将需要查询、排序、过滤或权限判断的核心字段放入 extend_shadow。
```

## 9. 错误码规则

---

### 9.1 错误码文件位置

每个 domain 的错误码定义文件位于：

```text
01_contracts/domains/<domain_id>/errors.yaml
```

示例：

```text
01_contracts/domains/ai-chatting/errors.yaml
```

---

### 9.2 错误码定义结构

错误码必须来自 S1 的业务规则或异常场景。

每个错误码必须同时包含稳定字符串 `code` 和数字 `value`。

推荐结构：

```yaml
- code: ERR_AI_CHAT_TOPIC_NOT_FOUND
  value: 110201
  http: 200
  cn: 话题不存在或不属于当前用户。
  en: Topic does not exist or does not belong to the current user.
  owning_domain: ai-chatting
  owning_module: topic
  range: 110200-110399
  retryable: false
  deprecated: false
  related_rules:
    - BR-AICHAT-001
  related_user_stories:
    - US-AICHAT-004
```

字段说明：

| 字段                   | 必填 | 说明                |
| -------------------- | -- | ----------------- |
| code                 | 是  | 稳定字符串错误码，使用大写蛇形命名 |
| value                | 是  | 稳定数字错误码           |
| http                 | 是  | HTTP 状态码          |
| cn                   | 是  | 中文错误提示            |
| en                   | 是  | 英文错误提示            |
| owning_domain        | 是  | 所属 domain_id      |
| owning_module        | 是  | 所属模块              |
| range                | 是  | 所属错误码区间           |
| retryable            | 是  | 是否可重试             |
| deprecated           | 是  | 是否已废弃             |
| related_rules        | 是  | 关联 S1 业务规则        |
| related_user_stories | 是  | 关联 S1 用户故事        |

---

### 9.3 错误码区间分配

S2 必须为每个 domain 下的每个模块预留一个连续的 `200` 值错误码区间。

错误码区间由全局错误码汇总文档统一登记，避免不同 domain 或模块之间发生冲突。

示例：

```text
110200 - 110399  ai-chatting / topic
110400 - 110599  ai-chatting / message
110600 - 110799  ai-chatting / assistant
110800 - 110999  ai-chatting / prompt-template
```

规则：

```text
每个模块默认占用 200 个连续错误码值。
同一模块内错误码必须落在该模块所属区间。
不同模块不得复用同一个错误码 value。
错误码 value 一旦 release，不得改变含义。
错误码 code 一旦 release，不得改变含义。
废弃错误码应标记 deprecated，不得复用。
如某模块 200 个错误码不足，必须新增区间，不得挤占其他模块区间。
```

---

### 9.4 HTTP 状态码规则

业务错误可以使用 HTTP `200` 返回，但必须通过响应体中的错误码判断业务结果。

适用场景：

```text
资源不存在但不希望暴露权限边界。
批量接口中的单项失败。
业务校验失败但请求本身已被服务端正常处理。
动作执行失败但 API 调用成功完成。
```

系统级错误仍可使用合适的 HTTP 状态码，例如：

```text
400 参数格式错误
401 未登录
403 无权限
404 路由不存在
500 服务端异常
```

具体 HTTP 状态码必须在 `errors.yaml` 中明确声明。

---

### 9.5 错误码汇总文档

S2 必须维护全局错误码汇总文档：

```text
01_contracts/error-code-index.md
```

该文件只负责登记错误码文件索引和错误码区间，不直接替代各 domain 的 `errors.yaml`。

推荐结构：

```markdown
# Error Code Index

本文档登记所有 domain 的错误码文件位置和错误码区间。

## 1. Domain 错误码文件索引

| Domain | 错误码文件 | 说明 |
|--------|------------|------|
| ai-chatting | `01_contracts/domains/ai-chatting/errors.yaml` | AI 聊天相关错误码 |
| model-management | `01_contracts/domains/model-management/errors.yaml` | 模型管理相关错误码 |

## 2. 错误码区间分配

| 区间 | Domain | Module | 说明 |
|------|--------|--------|------|
| 110200-110399 | ai-chatting | topic | 话题相关错误 |
| 110400-110599 | ai-chatting | message | 消息相关错误 |
| 110600-110799 | ai-chatting | assistant | 助手相关错误 |
| 110800-110999 | ai-chatting | prompt-template | 提示词模板相关错误 |
| 120200-120399 | model-management | provider | Provider 相关错误 |
| 120400-120599 | model-management | model | 模型相关错误 |

## 3. 分配规则

- 每个模块默认预留 200 个连续错误码。
- 新增 domain 或模块时，必须先在本文件登记区间。
- 新增错误码时，必须确认 value 落在已登记区间内。
- 已 release 的 value 不得复用。
- 废弃错误码必须在 domain errors.yaml 中标记 `deprecated: true`。
```

---

### 9.6 错误码校验要求

S2 校验时必须检查：

```text
所有 domain errors.yaml 均已被 error-code-index.md 索引。
所有错误码 value 都落在已登记区间内。
所有错误码 value 全局唯一。
所有错误码 code 全局唯一。
所有错误码都包含 related_rules 和 related_user_stories。
所有 related_rules / related_user_stories 引用真实存在。
所有 deprecated 错误码不得被新语义复用。
```

---

### 9.7 错误码禁止事项

禁止：

```text
不登记区间就新增错误码。
跨模块使用其他模块的错误码区间。
复用已 release 的错误码 value。
复用已 release 的错误码 code 表达新语义。
只写数字 value，不写字符串 code。
只写字符串 code，不写数字 value。
错误码缺少 S1 追溯引用。
让多个 domain 各自维护冲突的错误码区间。
```

## 10. 权限码规则

权限码必须来自 S1 的角色、可见性和操作规则。

文件位置：

```text
01_contracts/domains/<domain_id>/permissions.yaml
```

推荐结构：

```yaml
PERMISSION_CODE:
  module:
  resource:
  action:
  description:
  frontend_usage:
  backend_enforcement:
  related_rules:
    - BR-XXX-001
  related_user_stories:
    - US-XXX-001
```

要求：

```text
唯一
稳定
可用于展示控制
必须用于真实校验
必须关联 S1
```

## 11. 事件契约规则

事件定义必须来自 S1 的异步流程、任务状态、通知规则或跨模块协作需求。

文件位置：

```text
01_contracts/domains/<domain_id>/events.yaml
```

推荐结构：

```yaml
event_name:
  producer:
  consumers:
    - consumer_name
  trigger_condition:
  payload:
  idempotency_key:
  retry_policy:
  failure_policy:
  related_rules:
    - BR-XXX-001
  related_user_stories:
    - US-XXX-001
```

要求：

```text
事件名唯一
payload 明确
生产者明确
消费者明确
触发时机明确
幂等规则明确
失败处理明确
必须关联 S1
```

## 12. 模块契约规则

`module-contract.md` 用于定义模块职责和边界。

文件位置：

```text
01_contracts/domains/<domain_id>/module-contract.md
```

应包含：

```text
模块职责
模块不负责什么
输入
输出
依赖模块
被依赖模块
跨模块调用规则
数据归属
权限边界
事件边界
相关 S1 引用
```

## 14. S2 禁止事项

禁止：

```text
凭空新增 S1 未定义的核心业务能力。
与 S1 用户故事长期不一致。
将临时实现反向写为契约。
先实现后反推 OpenAPI。
将实际 migration 当作 S2 schema 事实源。
不经用户确认使用非 /api/v1 API 路径。
资源表缺少通用资源元数据字段且未说明原因。
错误码缺少区间登记。
错误码缺少 S1 追溯引用。
```

## 15. 最终规则

```text
S2 是实现契约事实源。
S2 必须从 S1 推导。
S2 必须可追溯到 S1。
S2 不得凭空新增 S1 未定义的核心业务能力。

API 必须统一使用 /api/v1。
请求和响应字段统一使用 lower_snake_case。
列表接口统一使用 page_num / page_size 等查询参数。
批量请求必须使用对象数组，不能直接传 ID 数组。
普通成功响应直接返回业务对象，不强制包装 code/data。
批量响应必须能逐项判断成功或失败。
资源表默认包含通用资源元数据字段。
错误码必须按 domain/module 区间分配并全局索引。
```
