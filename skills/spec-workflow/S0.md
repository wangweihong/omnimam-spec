# S0 探索阶段规则

S0 用于快速验证想法、交互、页面布局和功能可行性。

S0 不是事实源，但可以作为 S1 的输入。

---

## 1. 允许产出

S0 可以产出：

```text
可运行原型
页面 mock
交互 demo
流程草图
原型截图
原型说明
用户反馈
2. 禁止产出

S0 不允许产出正式契约或生产实现。

禁止产出：

正式 API
正式 SQL schema
正式错误码
正式权限码
正式事件契约
正式模块契约
用于生产环境的实现代码
3. 沉淀规则

用户确认后的 S0 原型必须沉淀到：

00_product/domains/<domain_id>/product-spec.md

并在 product-spec.md 的“原型来源”章节保留：

原型路径
原型目标
确认状态
沉淀范围
未沉淀内容

## 0. 原型来源

- 原型目标：

4. S0 与 S1 的关系
S0 可以作为 S1 输入。
S0 不能替代 S1。
S0 一旦沉淀为 S1，后续以 S1 为准。

如果 S0 与 S1 不一致：

以 S1 为准。
若 S0 中存在已确认但未沉淀的能力，必须补充到 S1。
5. S0 使用限制

禁止因为原型中存在某个字段、接口、流程，就直接把它作为正式契约。

原型中的能力必须经过 S1 沉淀，再由 S2 转化为契约。

正确链路：

S0 原型
  ↓ 用户确认
S1 product-spec.md
  ↓ 结构化
S2 openapi / schema / errors / permissions / events / module-contract
6. 最终规则
S0 是探索，不是事实源。
S0 只能输入 S1，不能跳过 S1 直接成为 S2 或正式实现依据。
