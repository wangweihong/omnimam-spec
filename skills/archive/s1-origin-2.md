# S1 产品语义事实源规则（修订版）

S1 是产品语义事实源，负责以**人类可顺序理解、可直接推导 S2** 的方式描述完整产品语义。

---

## 1. S1 定位

S1 回答：

```text
功能是什么，为什么需要，用户如何使用
系统提供哪些预设能力（种子数据）
核心业务对象是什么（含具体示例和配置契约）
对象间的关系与协作流程
业务规则与运行时行为（含算法步骤）
不同端的支持范围
如何验收
```

S1 不负责规定（留给 S2）：

```text
HTTP 路径与方法名
SQL 表结构、列名与索引
具体编程语言、框架
前端组件库与 CSS 类名
错误码数字编码
权限码实现细节
ORM 映射
```

但 S1 必须描述：

- 核心 API 的语义契约（请求/响应字段、含义、约束、示例），仅不绑定 HTTP 方法/路径；
- 领域服务/适配器的产品职责和协作关系；
- 运行时计算的步骤与决策逻辑。

## 2. 文件位置

全局 S1：

```text
00_product/glossary.md
00_product/global-business-rules.md
00_product/global-feature-matrix.md
```

领域 S1：

```text
00_product/domains/<domain_id>/product-spec.md
```

每个领域仅维护一份核心产品文档。内容过多时可拆分为主文档加受控子文档，但必须由主文档提供完整索引，保证一次性阅读的连贯性。

## 3. 文档组织原则

不强制固定模板。文档必须覆盖以下“内容清单”，但可使用领域最自然的结构（叙事、分步向导、参考式、示例驱动等）。

清单项（必须全部出现）：

- 文档信息与版本
- 原型来源（若基于 S0）
- 功能概述（目标、用户价值、范围）
- 预置数据与种子清单（若有）
- 核心业务对象与示例（含关系）
- 领域流程与运行时行为
- 业务规则（可集中或分散）
- 用户故事（含验收标准）
- 功能适配矩阵（端支持度）
- 非目标范围
- 待确认问题

验收标准可紧跟用户故事，也可集中成章，但必须显式编号并可直接验证。

## 4. 预置数据与种子清单

如果系统需要初始化能力分类、引擎类型、管理员可见的预设配置等，S1 必须明确给出这些具体数据，形式可以是列表、YAML、JSON 或结构化表格。

例如：

```yaml
capability_definitions:
  - code: video.text_to_video
    name: 文生视频
    inputs: [prompt, reference_image]
    outputs: [video]
  - code: image.upscale
    name: 图像放大
    inputs: [image, scale_factor]
    outputs: [image]
```

原则：这些数据是产品事实的一部分，写入 S1 不视为实现细节。

## 5. 核心业务对象与示例

允许并推荐使用 YAML/JSON 伪代码展示对象的完整结构、字段、约束和有效示例。只要不指定具体数据库类型或 ORM 注解即可。

对象描述应包含：

- 业务含义
- 逻辑字段、类型、约束
- 生命周期（创建、修改、删除、引用保护）
- 与其它对象的关系（含基数和引用规则）

关系可用表格或 Mermaid `erDiagram` 表达。

## 6. 领域流程与运行时行为

对以下场景，必须提供步骤化描述：

- 跨对象的端到端业务流程（如“导入工作流 → 发布应用 → 画布运行”）
- 运行时计算逻辑（如动态表单解析、约束传播、引擎选择）
- 异常路径与恢复策略

可使用：

- 编号步骤列表
- 流程图（`stateDiagram`、`flowchart`、`sequenceDiagram`）
- 伪代码（描述决策过程，不要求可编译）

示例：

```text
表单解析流程：
1. 读取 ApplicationVersion 及其关联的模板和能力绑定。
2. 根据当前已选字段值，过滤出仍有效的 CapabilityVariant。
3. 对每个字段：
   a. 从有效变体中提取允许值集合。
   b. 若当前值不在允许集合中，按字段策略（reset/clamp/reject）处理。
4. 返回字段、选项、修正信息及兼容引擎列表。
```

Mermaid 图应注明其补充说明的流程或规则 ID；如果图独立描述了完整流程，可给予流程编号（如 `PF-...`）并直接作为事实主体。

## 7. 业务规则

业务规则可集中编号（`BR-<DOMAIN_CODE>-<三位数字>`），也可直接嵌入相关章节。无论采用哪种方式，必须保证：

- 每条规则可被独立引用；
- 规则覆盖状态约束、权限、可见性、校验、默认值、删除保护、异步行为等；
- 规则描述中引用到对象时，使用 S1 定义的对象名和字段名。

## 8. 用户故事与验收标准

用户故事编号 `US-<DOMAIN_CODE>-<三位数字>`。每个故事包含：

- 一句话叙事（作为…我想要…以便…）
- 涉及角色、端
- 基本流程与异常
- 引用相关业务规则

验收标准编号 `AC-<DOMAIN_CODE>-<US三位数字>-<两位序号>`。每条验收标准应明确：

- 前置条件
- 用户操作步骤
- 预期结果（含界面反馈、数据变化、错误提示等）
- 适用端
- 异常覆盖

## 9. 功能适配矩阵

使用全局端类型。支持度标记：`✅` `❌` `⚠️` `🚧`。

矩阵中不应出现端类型之外的端标识。

## 10. 架构组件职责（领域服务）

若系统中存在不持久化但承担关键产品语义的架构组件（如适配器、执行器、解析器），S1 必须定义：

- 组件的产品职责（它完成什么业务功能）
- 与核心业务对象的关系
- 主要行为契约（可描述输入输出和步骤，但不写具体代码）

例如：

`EngineAdapter` 负责将标准化的平台操作请求转换为特定引擎的协议格式，包括鉴权、文件上传、任务提交与轮询。它不保存任何业务配置。

这能有效约束 S2 的模块划分。

## 11. 可视化规则

可使用 Mermaid 等图表辅助。图必须绑定到文本中明确存在的流程、规则或对象，不得引入文本中未定义的实体。若图表描述的是跨流程的综合视图，可将其作为独立的图形对象，并分配流程编号。

## 12. 聚合与可读性

所有图表、规则、故事应在同一文档或逻辑整体中可见，避免评审时频繁跨文件跳转。

优先使用线性叙事，减少纯编号网状引用。

复杂领域可保留单文件聚合，但若章节过多，允许拆分为主文档 + 子文档，由主文档提供完整索引和阅读路径。

## 13. 最终规则

S1 是产品语义唯一事实源。

S1 必须包含足够的具体事实（示例、配置、步骤），使得人类可以直接理解系统，AI 可以直接推导 S2。

S1 不规定实现细节，但通过契约示例和行为步骤指导实现。

S2 必须且只能从 S1 推导，所有偏离必须在 S2 中注明并回溯确认。
