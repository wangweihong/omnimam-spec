# S1 产品语义事实源规则

## 1. S1 定位

S1 是产品语义唯一事实源。

S1 负责以人类能够顺序阅读、完整理解，并能够直接推导 S2 实现契约的方式，描述一个领域的完整产品语义。

S1 不是表格、编号、用户故事或数据模型的集合，也不是用于压缩原始设计内容的摘要文档。

S1 的目标是：

> 在不规定具体技术实现的前提下，把产品实际是什么、系统中存在哪些事实、各对象如何协作、运行时如何工作、异常如何处理完整说明清楚。

对于平台型、基础设施型、编排型、执行型领域，S1 可以采用产品架构设计、领域机制说明、流程向导、配置参考、实例驱动等形式，不强制转换为传统 CRUD 产品需求模板。

---

## 2. S1 必须回答的问题

每个领域的 S1 必须能够回答以下问题。

### 2.1 产品与用户

```text
这个领域提供什么功能
为什么需要这个功能
解决哪些用户或系统问题
有哪些角色参与
用户从哪里进入
用户如何完成主要操作
用户最终得到什么结果
```

### 2.2 系统事实

```text
系统预置了哪些能力、类型、状态或配置
哪些数据由系统初始化
哪些组件由服务启动时注册
哪些数据由管理员维护
哪些数据由普通用户创建
哪些数据由运行时生成
哪些数据来源于外部系统
```

### 2.3 领域对象

```text
核心业务对象是什么
每个对象表达什么业务事实
对象属于哪一类
对象有哪些逻辑字段和约束
对象之间是什么关系
对象如何创建、修改、停用、归档或删除
哪些历史引用必须受到保护
```

### 2.4 领域运行机制

```text
多个对象如何协作完成完整业务流程
关键动作执行前检查什么
运行时如何计算、过滤、匹配和选择
异步任务如何提交、恢复、取消和完成
外部系统如何参与
失败后系统处于什么状态
最终结果如何进入其他领域
```

### 2.5 产品边界

```text
本领域拥有哪些业务对象
本领域不拥有哪些对象
依赖哪些其他领域
跨领域交互输入和输出是什么
哪个领域拥有最终状态事实
哪些能力本阶段不实现
```

### 2.6 验收

```text
如何证明主要功能已经正确实现
成功路径如何验收
异常路径如何验收
不同端支持哪些能力
哪些数据变化和状态变化必须被验证
```

## 3. S1 与 S2 的边界

### 3.1 S1 负责定义

S1 负责定义产品与领域事实，包括：

```text
产品目标
用户角色
业务能力
预置数据
领域对象
对象分类
对象关系
逻辑字段
字段业务约束
生命周期
状态语义
领域不变量
业务规则
关键校验
领域服务职责
运行时计算逻辑
跨对象业务流程
跨领域协作
外部系统交互语义
语义输入输出契约
典型配置实例
典型运行实例
失败与恢复策略
多端能力
用户故事
验收标准
非目标范围
```

### 3.2 S2 负责定义

以下内容由 S2 负责：

```text
HTTP 路径
HTTP 方法
URL 参数
HTTP Header
OpenAPI 结构
具体 DTO 名称
数据库表名
数据库列名
数据库类型
索引
外键实现
约束名称
错误码数字
权限码数字
消息队列主题
具体事件传输格式
具体编程语言
类名
函数名
接口方法签名
代码包结构
ORM 映射
框架选择
前端组件库
CSS 类名
状态管理库
具体 SDK 调用代码
```

### 3.3 S1 可以描述的契约

S1 必须描述核心能力的语义契约，但不绑定具体传输方式。

允许在 S1 中定义：

```text
服务或操作名称
调用目标
输入字段
输出字段
字段含义
字段约束
默认值
业务校验
状态变化
错误语义
调用步骤
示例请求
示例结果
```

例如，S1 可以定义“解析运行时表单”需要接收当前字段值并返回字段、合法候选值、修正项、违反项和兼容执行环境。

S1 不规定该能力最终通过 HTTP、内部调用、消息队列或其他方式实现。

### 3.4 回补规则

当 S2 无法从 S1 推导以下内容时：

```text
对象
状态
字段语义
业务规则
权限范围
校验条件
运行流程
跨领域责任
异常处理
```

不得由 S2 或代码自行决定。

必须先回补 S1，再继续推导 S2。

### 3.5 发现实现细节时的处理规则

审阅、整理或修改 S1 文档时，如果发现文档包含 3.2 节所列的任一 S2 实现细节，必须先保留原文并向用户询问处理指示。

该规则覆盖但不限于：

```text
HTTP 路径、HTTP 方法和其他传输细节
具体 Go 类型、interface、接口方法签名、函数名和包结构
前端组件、CSS 类名、状态管理库和其他前端实现细节
数据库、错误码、权限码、事件格式、框架和 SDK 调用等其他 S2 实现细节
```

询问用户时，必须说明：

```text
内容所在文件和位置
实现细节的类型
该内容与当前产品语义的关系及可能影响
可选择的处理方式
```

在收到用户明确指示前，不得：

```text
删除原文
修改或抽象化原文
迁移至 S2 或架构文档
仅记录到 review notes 或待确认问题后将其视为已处理
```

必须暂停涉及该内容的处理；与该内容无关的文档工作可以继续。

收到用户明确指示后，才可以按照指示将其保留为说明性示例、改写为产品语义、迁移至 S2 或架构文档，或者删除。

## 4. 文件位置与单文件原则

每个领域只维护一个 S1 核心产品文档：

```text
00_product/domains/<domain_id>/product-spec.md
```

一个领域的完整 S1 产品语义必须全部保存在该文件中。

禁止将领域 S1 拆分为：

```text
domain-model.md
business-rules.md
user-stories.md
flows.md
examples.md
acceptance.md
```

等多个子文档。

即使领域内容复杂、篇幅较长，也必须保持为一个 `product-spec.md`。

可以在 `product-spec.md` 内使用：

* 目录；
* 分篇；
* 分章；
* 附录；
* 折叠示例；
* 内部锚点；
* Mermaid 图；
* YAML、JSON 或伪代码块。

但不得通过拆文件破坏完整阅读路径。

### 4.1 全局文件引用

领域 S1 可以引用以下全局产品事实：

```text
00_product/glossary.md
00_product/global-business-rules.md
00_product/global-feature-matrix.md
```

全局文件只保存真正跨领域复用的事实。

领域特有的对象、流程、示例和规则必须保留在对应的 `product-spec.md` 中，不得为了减少篇幅转移到全局文件。

## 5. 文档组织原则

S1 不强制固定模板。

作者应根据领域性质选择最自然的组织形式，例如：

* 产品定位 → 领域架构 → 核心对象 → 流程 → 实例；
* 用户任务 → 对象模型 → 运行机制 → 异常处理；
* 能力目录 → 注册机制 → 配置流程 → 执行流程；
* 状态机 → 调度规则 → 运行恢复 → 验收；
* 端到端案例驱动，再反向解释对象与规则。

但无论采用何种结构，必须覆盖本规则要求的全部内容。

### 5.1 推荐阅读顺序

复杂平台型领域推荐按照以下顺序组织：

```text
文档信息
→ 产品定位与边界
→ 总体领域架构
→ 预置数据
→ 核心对象与组件
→ 对象关系
→ 创建与配置流程
→ 运行时流程
→ 典型端到端实例
→ 业务规则
→ 用户故事与验收
→ 多端能力
→ 非目标与待确认问题
```

该顺序是推荐结构，不是强制模板。

### 5.2 线性叙事优先

S1 应优先让读者顺序阅读后自然理解系统。

不得为了形式完整，将一条完整语义拆散到：

```text
实体表格
业务规则列表
用户故事列表
验收矩阵
追溯矩阵
```

并要求读者通过大量编号跳转才能重新拼接。

编号用于稳定引用，不得取代正文叙事。

## 6. S1 内容完整性清单

每份 `product-spec.md` 必须包含以下内容。

具体章节名称和顺序可以调整。

### 6.1 文档信息

至少包含：

```text
文档名称
版本
最后更新日期
作者或维护者
domain_id
domain_code
文档状态
```

### 6.2 原型来源

若来源于 S0，必须说明：

```text
原型路径
原型目标
原型确认状态
已沉淀内容
未沉淀内容
已经废弃的原型方案
```

若没有可确认原型，应明确声明。

### 6.3 功能概述

必须说明：

```text
领域目标
用户价值
适用角色
适用范围
主要能力
与其他领域的关系
本阶段重点
```

### 6.4 预置数据与种子清单

存在系统预置事实时必须给出具体清单。

### 6.5 核心业务对象和架构组件

必须说明业务对象、对象分类、字段、约束、关系以及关键非持久化组件。

### 6.6 领域流程与运行时行为

必须描述主要配置流程、执行流程、跨对象流程、算法步骤和失败处理。

### 6.7 业务规则

规则可以集中或分散在相关章节，但必须可独立引用。

### 6.8 典型实例

复杂领域必须包含完整配置实例、完整运行实例和至少一个失败实例。

### 6.9 用户故事与验收标准

必须覆盖主要角色和主要业务闭环。

### 6.10 功能适配矩阵

必须说明不同端的支持范围。

### 6.11 非目标范围

必须明确本阶段不实现的能力。

### 6.12 待确认问题

必须集中记录仍未确认且可能影响 S2 的产品问题。

## 7. 预置数据与种子清单规则

如果系统需要初始化以下内容，S1 必须明确给出实际数据：

```text
能力定义
引擎类型
对象类型
状态类型
端口类型
媒体类型
策略类型
默认角色
内置操作
内置转换类型
系统默认配置
```

这些数据属于产品事实，不属于实现细节。

不得只声明：

> 系统初始化部分能力定义。

必须列出具体内容。

示例：

```yaml
capability_definitions:
  - code: video.text_to_video
    name: 文生视频
    inputs:
      - name: prompt
        type: string
        required: true
    outputs:
      - name: video
        type: asset.video

  - code: image.upscale
    name: 图像放大
    inputs:
      - name: image
        type: asset.image
        required: true
      - name: scale_factor
        type: number
        required: true
    outputs:
      - name: image
        type: asset.image
```

预置数据必须说明：

```text
数据来源
由谁维护
是否允许管理员修改
是否允许停用
系统升级时如何处理
是否允许删除
冲突时以什么为准
```

## 8. 对象分类规则

每个核心对象必须明确其对象类别，防止持久化实体、运行时对象和代码注册组件相互混淆。

允许使用以下分类。

### 8.1 产品事实对象

表示平台长期维护的业务事实。

例如：

```text
CapabilityDefinition
ProviderCapability
Application
ApplicationVersion
```

### 8.2 配置对象

表示管理员或用户为运行系统配置的数据。

例如：

```text
ApplicationEngineInstance
EngineCapabilityBinding
ApplicationTemplate
```

### 8.3 运行对象

表示一次执行、任务或业务运行过程。

例如：

```text
ApplicationRun
CanvasRun
Artifact
```

### 8.4 派生对象

由其他事实和当前上下文计算得到，不应作为新的独立事实源。

例如：

```text
RuntimeFormSchema
EffectiveCapability
CompatibleEngineSet
```

### 8.5 外部引用对象

表示本领域引用但不拥有的其他领域对象。

例如：

```text
TaskRun
Asset
Canvas
User
```

### 8.6 系统注册组件

不作为业务数据由用户维护，而是由服务代码、模块或插件在启动时注册。

例如：

```text
ApplicationEngineType
EngineAdapter
OperationExecutor
RuntimeFormResolver
```

### 8.7 对象分类最低要求

每个核心对象必须说明：

```text
对象类别
业务定位
是否持久化
数据来源
创建者
维护者
生命周期
负责什么
不负责什么
```

## 9. 核心业务对象规则

S1 可以并推荐使用 YAML、JSON、结构化表格或伪代码描述对象。

不得因为对象复杂而仅压缩成一行实体表格。

每个核心对象至少必须包含以下内容。

### 9.1 业务定位

说明：

```text
对象是什么
对象解决什么问题
对象代表什么业务事实
对象不是什么
```

### 9.2 逻辑字段

允许描述：

```text
字段名称
逻辑类型
是否必填
默认值
业务含义
取值约束
逻辑唯一性
引用对象
是否可修改
是否进入历史快照
```

允许使用的逻辑类型示例：

```text
string
integer
number
boolean
enum
date
date-time
duration
money
asset.image
asset.video
asset.audio
object
object-list
reference
reference-list
schema
expression
```

禁止在 S1 中定义：

```text
数据库列名
数据库字段类型
主键类型
索引名
外键名
ORM tag
JSON 序列化 tag
具体语言 struct
```

### 9.3 生命周期

必须说明：

```text
如何创建
创建前检查什么
谁可以修改
哪些字段不可修改
何时启用
何时停用
何时归档
何时允许物理删除
哪些历史引用受到保护
```

### 9.4 数据来源

必须说明：

```text
SYSTEM_SEED
SYSTEM_REGISTRATION
ADMIN
USER
RUNTIME
DERIVED
EXTERNAL_SYNC
```

中的一种或多种来源，并解释来源优先级。

### 9.5 完整示例

复杂对象必须至少提供一个有效实例。

示例必须足以让读者理解对象中的字段如何共同表达一项业务事实。

## 10. 对象关系规则

涉及两个及以上核心对象时，必须明确描述对象关系。

不得只在字段中出现某个 `xxx_id`，而不解释关系语义。

关系必须说明：

```text
关系双方
关系名称
关系方向
关系基数
关系成立条件
关系由谁创建
是否允许为空
是否允许重复
状态变化的影响
停用影响
删除影响
历史引用保护
```

允许的业务关系包括：

```text
归属
聚合
组合
引用
绑定
声明支持
实例化
派生
发布
执行依赖
运行关联
产生
登记
```

示例：

```text
ApplicationEngineType 声明支持多个 CapabilityDefinition。

一个 CapabilityDefinition 可以由多个 ApplicationEngineType 支持。

ApplicationEngineType 只有在对应能力的全部必要 OperationExecutor
均已注册时，才能把该能力声明为可用。
```

这种关系属于产品语义，不是数据库外键实现。

## 11. 架构组件职责规则

某些对象不持久化，但承担关键产品语义和运行责任。

满足以下条件之一的组件应进入 S1：

```text
跨多个业务对象协作
决定关键业务执行路径
承担外部系统适配
承担运行时解析或调度
其职责边界会影响 S2 模块设计
缺少该组件会导致产品行为无法解释
```

典型组件包括：

```text
EngineAdapter
OperationExecutor
ApplicationExecutor
RuntimeFormResolver
WorkflowParser
EngineSelector
ArtifactRegistrar
```

每个组件必须说明：

```text
产品职责
输入语义
输出语义
使用哪些核心对象
产生或修改哪些对象
调用哪些外部系统
失败时返回什么业务结果
不负责什么
```

S1 不应描述普通 CRUD Service、Repository、Controller 或工具函数。

### 11.1 组件契约示例

```text
EngineAdapter

职责：
负责某类执行平台的连接、鉴权、公共文件上传、健康检查、
通用错误归一和平台级请求处理。

输入：
ApplicationEngineInstance 和平台级操作上下文。

输出：
平台连接结果、上传引用、健康状态或标准化失败原因。

不负责：
模型清单、模型生命周期、能力参数范围和用户应用表单。
```

## 12. 领域服务语义契约

核心领域服务或操作必须定义语义契约。

推荐格式：

```yaml
operation: ResolveRuntimeForm

purpose: 根据应用版本、当前输入和执行环境计算最终可用表单

input:
  application_version:
    type: reference
    required: true
  current_values:
    type: object
    required: true
  engine_context:
    type: object
    required: false

output:
  fields:
    type: object-list
  compatible_engines:
    type: reference-list
  corrections:
    type: object-list
  violations:
    type: object-list

rules:
  - 只保留满足全部约束的有效能力变体
  - 提交运行时必须重新解析
```

该契约不规定：

```text
HTTP 路径
HTTP 方法
具体 DTO 类名
序列化方式
```

## 13. 领域流程规则

跨对象业务行为必须通过步骤化流程描述。

领域流程编号：

```text
PF-<DOMAIN_CODE>-<三位数字>
```

示例：

```text
PF-AIAPP-001
PF-TASK-004
```

### 13.1 必须编号的流程

以下流程应分配 PF 编号：

```text
系统初始化
自动注册
管理员配置
对象创建
绑定
导入
发布
执行
异步处理
取消

恢复
```

```text

异步行为
失败分支

```

### 13.3 对象变化必须明确

不得只写：

```text

```

必须明确：

```text

什么时候创建应用版本
失败时哪些对象不得创建

```

### 13.4 流程不是代码调用顺序

S1 描述业务步骤，而不是具体函数调用。

允许：

```text

系统验证工作流中的节点类型和参数。
系统生成可执行工作流表达。
```

不应写：

```text

```

## 14. 运行时计算与算法步骤规则

凡是涉及动态计算、过滤、选择、约束传播、状态投影、调度或恢复的逻辑，必须写出步骤。

不得使用以下模糊描述：

```text
系统智能匹配
系统自动处理
系统校验合法性
系统选择合适引擎
系统解析表单
```

必须展开决策过程。

示例：

```text
RuntimeFormSchema 解析流程：

1. 读取 ApplicationVersion。
2. 读取关联的 ApplicationTemplate。
3. 读取允许使用的能力和执行环境。
4. 取得所有候选 CapabilityVariant。
5. 应用 EngineCapabilityBinding 的限制。
6. 应用 ApplicationTemplate 的固定值、白名单和附加约束。
7. 使用当前字段值过滤无效 Variant。
8. 从剩余 Variant 中计算每个字段的合法候选值。
9. 当前值失效时，按照 reset、fallback、clamp 或 reject 处理。
10. 返回字段、候选值、修正项、违反项和兼容执行环境。
```

### 14.1 算法描述要求

算法步骤必须说明：

```text
输入事实
过滤顺序
优先级
冲突处理
默认行为
无结果时的处理
结果对象
是否需要在提交时重新计算
```

### 14.2 伪代码

可以使用不可编译伪代码增强理解：

```text
validVariants = allVariants
validVariants = filterByEngineBinding(validVariants)
validVariants = filterByTemplateConstraints(validVariants)
validVariants = filterByCurrentValues(validVariants)

if validVariants is empty:
    return violation(NO_VALID_VARIANT)
```

伪代码属于行为说明，不视为具体实现代码。

## 15. 关键动作校验规则

涉及以下动作时，必须明确列出服务端业务校验：

```text

停用
归档
```

```text
- 不保留部分结果；
```

```text
主要流程是否有用户故事
主要成功和失败路径是否有验收标准
验收是否覆盖对象、关系和状态变化
```

---

## 31. S1 不合格条件

出现以下任一情况时，不得进入 S2 推导：

```text
只列实体字段，没有解释系统如何工作
只列业务规则，没有连续流程
只列用户故事，没有领域运行机制
只列抽象对象，没有实际配置示例
声明存在种子数据，但没有列出具体内容
存在多种对象类型，但没有说明行为差异
存在 Adapter 或 Executor，但没有说明职责
关键动作只写“校验合法性”
运行时逻辑只写“系统智能匹配”
没有说明对象创建和状态变化
没有完整端到端实例
没有失败和恢复实例
没有跨领域责任边界
ComfyUI 流程只写“上传并解析工作流”
SaaS 流程没有能力注册、绑定和执行链路
文档必须依赖大量编号跳转才能理解
为了符合模板而删除原始设计细节
S2 仍需自行发明关键产品语义
```

---

## 32. 最终规则

1. S1 是产品语义唯一事实源。

2. 每个领域只维护一个：

```text
00_product/domains/<domain_id>/product-spec.md
```

3. S1 不强制固定模板，但必须完整覆盖产品事实、对象、组件、关系、流程、实例、规则、跨领域边界和验收。

4. S1 不以压缩文档为目标，不得为了套用实体表格、编号体系或用户故事模板而删除原始设计细节。

5. S1 可以并应当包含：

```text
预置数据
配置契约
对象完整示例
服务语义契约
运行算法步骤
领域组件职责
外部交互语义
端到端业务流程
失败和恢复策略
```

6. S1 不规定具体代码和传输实现，但必须将产品运行机制描述到不需要开发者自行猜测。

7. 编号、矩阵和图表用于提高引用与追溯能力，不得取代完整、连续、可理解的正文。

8. 复杂领域应优先通过真实实例验证抽象模型，而不是继续增加抽象术语。

9. S2 必须且只能从 S1 推导。

10. 当 S2 发现 S1 缺失产品语义时，必须回补 `product-spec.md`，不得在 S2 或代码中自行补全。

S1 的最终质量标准是：

> 人类顺序阅读 `product-spec.md` 后能够理解产品实际如何工作；AI 或开发人员能够据此推导 S2，而不需要自行猜测核心业务逻辑。
